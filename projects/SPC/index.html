Highlights

建立 I-MR、X̄-R、EWMA 與 CUSUM 管制圖，並整合 Nelson 8 條判異規則，偵測製程水準位移、趨勢、週期與組內穩定度異常。

針對長晶連續訊號的高度自相關問題，以合理子組建立 X̄-R 管制圖，避免直接使用相鄰樣本導致組內變異被低估及管制界線失真。

整合 Cp、Cpk、Pp、Ppk、預估不良 ppm、Shapiro-Wilk 常態檢定與 Box-Cox 轉換，避免在非常態資料上直接產生誤導性的製程能力指標。

提供自相關診斷、AR(1) 殘差管制圖、管制界線信賴區間與 ARL₀ 評估，將統計假設及基準資料不足造成的不確定性呈現在操作介面。

使用 Hotelling T² 整合六項品質特性，搭配變數貢獻度與相關矩陣，識別單一指標仍在界內、但多變量聯合狀態已經偏移的異常。

建立工程規格的提議、簽核、駁回與版本歷史流程，採用 PE／QA 部門、Engineer／Lead 職級及跨部門簽核規則，避免提議者自行核准規格。

透過 POST /spc/ingest 接收即時資料，使用 Redis Pub/Sub 支援多 Worker 廣播，再以 SSE 將 Phase II 監控結果推送至瀏覽器；Redis 不可用時可降級至記憶體廣播。

整合 OOC 告警、事件處置狀態、嚴重度分級、SLA 逾期追蹤、Email 去重與沉默期，避免製程失控期間產生重複通知或告警轟炸。

使用 APScheduler 執行週期性報表任務，透過 ReportLab 產生內嵌中文字型的 PDF 報表，並支援 Email 寄送及 CSV 分析結果匯出。

將唯讀製程分析資料與平台可寫狀態分離：Parquet 資料可由本機或 Google Cloud Storage 讀取；工程規格、事件註記、使用者、基準與即時資料則獨立保存。

Engineering Focus

此專案不只是呈現 SPC 圖表，而是展示如何建立具備統計有效性、即時監控與製程治理能力的完整品質管理平台。

設計重點包含合理子組設計、自相關修正、非常態製程能力、管制界線不確定性、多變量異常偵測、Phase I／Phase II 分離、跨部門規格簽核、告警生命週期，以及唯讀研究資料與平台狀態的責任邊界。

平台採用 Vue 3、ECharts 與 Pinia 建立互動式分析介面；後端使用 FastAPI、Pydantic、NumPy、SciPy 與 pandas 實作統計分析服務，並透過 PyArrow、fsspec 與 gcsfs 統一存取本機及 GCS Parquet 資料。

正式環境以 Docker Compose 編排 FastAPI 與 Redis，由 Hetzner 主機 Nginx 提供 HTTPS、前端靜態資源及 /api 反向代理，後端僅綁定本機介面，降低服務直接暴露的風險。

本平台使用模擬及研究資料，不連接實體 PLC，也不執行生產設備控制。
